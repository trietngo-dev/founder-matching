# 業務要件一覧

| id | category | requirement | priority | description |
| --- | --- | --- | --- | --- |
| BR-001 | プロフィール管理 | 学生アカウントの初期登録およびログイン | 高 | 授業を受講する学生が本プラットフォームを利用開始するための初期登録およびログインプロセス。※要確認：大学の既存LMS（学習管理システム）や学内共通認証（SSO）との連携有無、または専用アカウント作成機能の要否。 |
| BR-002 | プロフィール管理 | 専攻・基本所属情報の登録 | 高 | 学生が自身の所属（学部、学科、専攻分野）および基本情報（学年など）を登録・管理するプロセス。※要確認：専攻マスターの選択形式や複数専攻を保持する場合の登録方法。 |
| BR-003 | プロフィール管理 | ハードスキルおよびソフトスキルの登録・自己評価 | 高 | AIによるスキル補完マッチングのために、学生が自身の保有するスキル（技術、デザイン、営業、財務などのハードスキル、および協調性、実行力などのソフトスキル）を選択し、その習熟レベルを登録するプロセス。※要確認：スキル評価の判定基準・評価尺度の統一ルール。 |
| BR-004 | プロフィール管理 | 起業モチベーションおよびコミットメント目標の設定 | 高 | 学生が本プログラムにかける熱量（「授業を単位通過させたい」から「本格的にスタートアップとして起業したい」まで）や、プロジェクトに週あたり費やせる稼働可能時間、目標の整合性を登録するプロセス。※要確認：マッチングアルゴリズムへの影響比重および目標情報の非公開希望機能。 |
| BR-005 | プロフィール管理 | 初期役割志向（Idea Owner / Talent）の設定 | 高 | 自身の起業アイデアでメンバーを募る側（Idea Owner）か、自身の専門性を活かして他者のアイデアに参画する側（Talent）か、マッチングにおける基本志向を設定するプロセス。 |
| BR-006 | プロフィール管理 | プロフィール情報の随時更新と維持 | 中 | 講義の進捗やスキルの習得、熱量の変化に伴い、学生自身がいつでもプロフィール情報を最新に維持・更新できるプロセス。※要確認：チームマッチング成立後におけるプロフィール更新（特にチーム全体のスキルバランス計算に影響する項目）の制限ルール。 |
| BR-007 | プロフィール管理 | プロフィール公開設定および通知の管理 | 中 | 他学生（候補者）への自プロフィールの公開可否や、マッチング希望受信時の連絡通知設定（メール等）を行うプロセス。※要確認：学内の個人情報保護方針に準拠した連絡先情報（メールアドレス等）の表示・非表示のビジネスルール。 |
| BR-008 | プロフィール管理 | プロフィール登録ステータス（マッチング準備完了）の判定 | 高 | プロフィールに入力不備（必須項目未入力など）がある場合、AI推薦やマッチング探索に参加できないよう「未完了」ステータスとして扱い、必須項目を埋めるよう促すプロセス。※要確認：必須となるプロフィールの項目項目群の定義。 |
| BR-009 | プロジェクト・アイデア管理 | 起業アイデア・プロジェクトの新規登録プロセス | 高 | アイデア保持者（Idea Owner）が、新規の起業アイデア（プロジェクト名、ビジョン、概要、事業化の目的など）をシステムに登録し、構成メンバーを募集できる初期状態にする業務プロセス。下書き保存機能も含む。 |
| BR-010 | プロジェクト・アイデア管理 | 募集ポジション（求める役割とスキル）の設定プロセス | 高 | アイデア保持者（Idea Owner）が、チームに必要な役割（例：エンジニア、デザイナー、マーケターなど）と、それぞれの役割で求める具体的なスキル（ハードスキル/ソフトスキル、求める専門分野）を定義・設定する業務プロセス。 |
| BR-011 | プロジェクト・アイデア管理 | プロジェクト公開プロセスの承認/自動公開処理 | 中 | 起案されたプロジェクトをプラットフォーム上に公開し、他の学生が閲覧・応募できる状態にする。※要確認：教員や管理者による内容確認と事前承認プロセスを挟む必要があるか、あるいは登録即時自動公開とするか、運用ルールを定義する必要がある。 |
| BR-012 | プロジェクト・アイデア管理 | プロジェクト情報の変更・ブラッシュアッププロセス | 高 | プロジェクト推進中、アイデアのピボット（方向転換）や募集要件の変更、チームの活動状況に応じて、プロジェクトの詳細内容や募集ポジションの情報を、アイデア保持者が適宜更新・メンテナンスする業務プロセス。 |
| BR-013 | プロジェクト・アイデア管理 | 募集ステータス（募集中/募集停止/成立）の変更管理プロセス | 高 | メンバーの応募状況、選考・マッチングの進捗、またはチーム結成の完了（Founding Team成立）に伴い、アイデア保持者がプロジェクトの募集ステータスを切り替える業務プロセス。※要確認：必要メンバーが全員揃った段階で自動的に募集終了ステータスへと移行させるかどうかのビジネスルール。 |
| BR-014 | プロジェクト・アイデア管理 | プロジェクトの中止・アーカイブプロセス | 中 | 何らかの理由でプロジェクトが立ち消えとなった場合や、授業の期が終了した場合に、プロジェクトデータを削除せず「アーカイブ（過去ログ）」状態に変更するプロセス。※要確認：学生自身によるアーカイブ操作権限の範囲、および教員による強制アーカイブ機能の要否。 |
| BR-015 | マッチング・推奨管理 | AIによる候補メンバー推薦プロセス | 高 | アイデア保持者がプロジェクトを起稿した際、AIがプロジェクト要件と登録学生のプロファイル（スキル、専攻、興味等）を分析し、バランスを考慮して最適な推薦メンバーを提示する業務プロセス。※AIの推薦アルゴリズムの適用ルールや基準はビジネスルール未提供のため要確認。 |
| BR-016 | マッチング・推奨管理 | AIによる候補プロジェクト推薦プロセス | 高 | 特定のスキルを持つ学生（Talent）に対し、本人のスキル、専攻、興味分野と補完関係にあり、現在メンバーを募集している適切なプロジェクトをAIが自動推薦・提示する業務プロセス。※推薦スコアの表示基準は技術制約未提供のため要確認。 |
| BR-017 | マッチング・推奨管理 | 条件指定による手動メンバー検索プロセス | 高 | アイデア保持者が、特定のハードスキル・ソフトスキル、専門分野、コミットメントレベルなどの条件を手動で指定し、要件に合致するメンバーを能動的に検索・リストアップする業務プロセス。 |
| BR-018 | マッチング・推奨管理 | 条件指定による手動プロジェクト検索プロセス | 高 | スキル保有者が、自身のスキルを活かせるプロジェクトや、興味のある業界、必要とする役割、活動予定時間などを手動で指定し、適切なプロジェクトを検索・リストアップする業務プロセス。 |
| BR-019 | マッチング・推奨管理 | チーム内のスキル補完状況可視化プロセス | 中 | 特定の候補者をチームに誘う際、現在のチームが持つ保有スキルと候補者のスキルを重ね合わせ、チーム全体のスキルカバー率や不足している専門領域をグラフ等でビジュアル分析する業務プロセス。※可視化のデザインや出力ルールは技術要件未提供のため要確認。 |
| BR-020 | マッチング・推奨管理 | 目標・コミットメント不一致アラートプロセス | 高 | マッチング交渉を行う前に、双方の「起業に対する熱量」や「目標（単位取得目的か、実際の起業目的か）」、「稼働可能時間」に大幅なズレがある場合、システムが自動的に不一致を検出し、交渉前に対話を促す警告を表示する業務プロセス。※警告を出す不一致の判定基準はビジネスルール未提供のため要確認。 |
| BR-021 | マッチング・推奨管理 | 候補メンバー・プロジェクトのお気に入り登録・管理 | 中 | 探索中に見つけた興味のあるメンバーやプロジェクトを「お気に入り」として一時保存し、複数の候補を比較検討した上でオファーや応募を決定する業務プロセス。 |
| BR-022 | マッチング・推奨管理 | マッチング適合度（スコア）算出プロセス | 中 | 各メンバーやプロジェクトの要件適合率（％）をシステムが自動算出し、検索結果や推薦リスト上にスコア表示することで、意思決定を容易にする業務プロセス。※適合率の算出ロジックはドメイン知識・技術要件未提供のため要確認。 |
| BR-023 | マッチング・推奨管理 | ロール競合・重複検知プロセス | 中 | チーム内で同じ役割（例：ビジネスリーダーが複数名など）が重複し、バランスが崩れるようなチーム構成になりかけた際に、システムが自動的に役割の競合を検出し、確認メッセージを表示する業務プロセス。※重複制限をかける役割の定義はビジネスルール未提供のため要確認。 |
| BR-024 | マッチング・推奨管理 | AI推薦パラメータのモニタリングと調整プロセス | 低 | システム管理者が、過去の結成チームの成功パターンやマッチング成立データ（蓄積データ）を基に、AI推薦アルゴリズムの重視パラメータ（スキル優先か、コミットメント優先か等）の重み付けを定期的に見直し・微調整する管理運用プロセス。※管理画面の仕様や調整運用手順は運用・技術要件未提供のため要確認。 |
| BR-025 | チーム結成・合意フロー管理 | プロジェクト起案者から特定学生への加入オファー送信 | 高 | 起業アイデアを提示しているプロジェクトオーナー（Idea Owner）が、求めるスキルや目標が合致する未所属の学生（Talent）に対し、チームへの加入オファーを送信できる。オファー送信時には、期待する役割やメッセージを付与できるものとする。 |
| BR-026 | チーム結成・合意フロー管理 | 学生からプロジェクトへの参画希望申請 | 高 | 特定のプロジェクトに興味を持った未所属の学生が、プロジェクトオーナーに対して参画希望の申請（応募）を送信できる。申請時には、自身の志望動機や貢献できるスキルについてのメッセージを付与できるものとする。 |
| BR-027 | チーム結成・合意フロー管理 | オファー・参画申請に対する諾否回答 | 高 | オファーまたは参画希望申請を受け取った側（学生またはプロジェクトオーナー）は、内容を確認し、受託（承認）または辞退を選択して回答する。辞退時には、任意で理由をフィードバックできるものとする。 |
| BR-028 | チーム結成・合意フロー管理 | 双方向合意（仮マッチング）および連絡調整 | 高 | 双方の合意（オファー承認または申請承認）が得られた場合、ステータスを「仮マッチング」とし、チーム結成に向けた詳細な相互確認のためのコミュニケーション（メッセージ送受信や面談日程調整など）を可能とする。 |
| BR-029 | チーム結成・合意フロー管理 | コミットメントおよび役割の最終確認（※要確認） | 中 | 仮マッチング後、本結成に進む前に、授業外の活動時間、目標（単位取得のみか起業化か）、役割分担等の合意事項を確認・登録する（※目標の不一致を防ぐための確認プロセスやビジネスルール詳細は要確認）。 |
| BR-030 | チーム結成・合意フロー管理 | チーム結成の最終決定とロック | 高 | プロジェクトオーナーは、必要なメンバーが揃い合意が取れた段階で、システム上で「チーム結成確定」の操作を行う。確定により、チーム構成員が固定（ロック）される。 |
| BR-031 | チーム結成・合意フロー管理 | 結成確定に伴う他案件の自動キャンセルとステータス更新 | 高 | チームが最終確定した際、所属となった全メンバーのステータスを「所属済」に変更し、他のプロジェクトに送信していたオファーや参画申請、および他から受信していた未回答のオファー等をシステム側で自動的にキャンセル（取り消し）する。 |
| BR-032 | チーム結成・合意フロー管理 | 教員へのチーム結成報告および承認プロセス（※要確認） | 中 | 最終確定したチーム情報は、授業の担当教員へ自動的に報告・提出される。教員の管理画面での「承認」をもって正式なクラス内チームとして認可される業務プロセスとする（※教員による承認フローの有無や運用手順については要確認）。 |
| BR-033 | クラス・運営管理 | 受講学生の登録および受講クラス割り当て業務 | 高 | 担当教員または運営管理者は、起業科目の受講学生データを登録し、該当する授業クラスに割り当てる。※既存の学内LMSや学務システムとのアカウント・受講情報の自動連携の有無については未定義のため要確認。 |
| BR-034 | クラス・運営管理 | 授業クラス設定およびマッチング期限管理業務 | 高 | 担当教員は、授業クラスごとにチーム結成プロセスを開始し、プロファイルの登録期限やマッチング（オファー送信・合意）の締め切り日時を設定・変更する。 |
| BR-035 | クラス・運営管理 | チーム結成ステータスのリアルタイム監視業務 | 高 | 担当教員および運営管理者は、管理画面を通じて学生のチーム結成ステータス（チーム結成完了、交渉中、未マッチングなど）や、各チームのプロジェクト登録状況をリアルタイムで監視する。 |
| BR-036 | クラス・運営管理 | チーム結成状況のレポート出力業務 | 中 | 担当教員は、授業での成績評価や進捗管理を円滑に行うため、結成されたチームの一覧、所属する学生の情報、プロジェクト名などをレポート（CSV等のファイル）として出力する。※レポートに必要なデータ項目の詳細仕様については要確認。 |
| BR-037 | クラス・運営管理 | 期限超過者・未結成学生への教員介入・強制マッチング業務 | 高 | マッチング期限を過ぎてもチームが結成されていない学生や、調整が難航している学生に対し、担当教員が介入し、システム上で特定のチームへの強制的な割り当てや、新しいチームの直接編成を実施して最終確定させる。※手動編成におけるルール制限については要確認。 |
| BR-038 | クラス・運営管理 | プラットフォームの稼働サイクルおよびデータクリーンアップ業務 | 中 | 運営管理者は、学期やプログラムが終了した際、プラットフォーム全体の稼働状況や利用統計を確認し、次の期に向けたデータのクリーンアップ、または過去データのアーカイブ処理を行う。※長期的なデータ保管に関する運用・セキュリティルールは未定義のため要確認。 |
